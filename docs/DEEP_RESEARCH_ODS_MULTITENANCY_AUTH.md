# Deep Research: ODS, Multi-tenancy и вливание аутентификации

Документ для консультации (в т.ч. с Мишей): как понимать ODS в контексте RAGfun, текущая мультитенантность и что даёт вливание системы аутентификации в основную ветку.

---

## 1. Что такое ODS в нашем контексте

**ODS (Operational Data Store)** — операционное хранилище данных: слой, который:

- **интегрирует** данные из нескольких источников в одном месте;
- даёт **актуальный снимок** (near-real-time), а не долгую историю;
- используется для **операционных решений** и отчётности, без прямой нагрузки на транзакционные системы.

В RAGfun роль ODS-подобного слоя выполняют:

| Компонент | Роль в духе ODS |
|-----------|------------------|
| **document-storage** | Единая точка хранения метаданных документов (Postgres) и файлов (S3). Принимает данные от Gate, отдаёт их doc-processor и retrieval. |
| **Postgres (document_metadata)** | Текущее состояние документов: doc_id, tenant_id, project_id, source, tags, acl, ingestion state. По сути — операционный снимок «что сейчас есть». |
| **Gate** | Оркестратор: собирает запросы от UI/клиентов, стучится в storage и retrieval. Не хранит состояние, но задаёт потоки данных. |

То есть: **ODS у нас — это не отдельный продукт, а архитектурная роль**: «единый операционный слой документов и метаданных» (document-storage + Postgres), от которого питаются индексация (retrieval, OpenSearch, Qdrant) и поиск/чат.

### Какой вариант ODS нам подходит

В практике мультитенантных ODS обычно различают два подхода:

| Вариант | Описание | Плюсы | Минусы |
|--------|----------|--------|--------|
| **Shared ODS** | Одна БД, одна схема, колонка `tenant_id`, фильтрация в каждом запросе | Один деплой, проще бэкапы и масштабирование, проще разработка | Нужна жёсткая дисциплина: везде фильтр по tenant_id; после вливания auth — проверка доступа в Gate |
| **Tenant-isolated ODS** | Отдельная БД или схема на каждого тенанта | Максимальная изоляция, удобно при жёстких регуляторных требованиях | Много инстансов, сложнее операции и обновления |

**Рекомендация для RAGfun: вариант Shared ODS** (единый ODS с `tenant_id`).

- У вас он уже по факту реализован: общий Postgres, общие индексы OpenSearch/Qdrant, везде проброс и фильтр по `tenant_id`.
- Один стек (document-storage, retrieval, Gate) проще сопровождать и масштабировать горизонтально.
- Изоляция достигается за счёт **обязательной фильтрации по tenant_id** и, после вливания auth, **вывода/проверки tenant_id в Gate по токену**, а не за счёт отдельных БД на тенанта.

Переход на tenant-isolated ODS (отдельная БД/схема на тенанта) имеет смысл только при явных требованиях: регуляторика, контракты с разными юрисдикциями или настолько разные объёмы тенантов, что выносить их в отдельные кластеры выгоднее. Пока таких требований нет — остаёмся на shared ODS.

## 2. Multi-tenancy в RAGfun сегодня

### 2.1 Как устроено

- **tenant_id** — опциональное поле во всех ключевых точках:
  - **Gate**: загрузка (`Form`), чат (`filters.tenant_id`), коллекции (`?tenant_id=`);
  - **document-storage**: метаданные в Postgres, индексы по `tenant_id`, листинг коллекций по tenant;
  - **retrieval**: фильтр по `tenant_id` (OpenSearch term, Qdrant filter);
  - **doc-processor**: проброс `tenant_id` из документа в чанки;
  - **agent-search**: опциональный `tenant_id` в запросах.

- **Нет привязки к пользователю**: `tenant_id` передаётся явно (форма, query, body). Кто угодно, кто знает tenant_id, может читать/писать данные этого тенанта.

### 2.2 Связь с ODS

В классическом ODS мультитенантность часто реализуют так:

- отдельные БД/схемы на тенанта, или
- общая схема с колонкой `tenant_id` и строгая фильтрация по ней.

У нас — второй вариант: общая схема, везде пробрасывается `tenant_id`, фильтрация в запросах и индексах. То есть **мультитенантность уже встроена в ODS-слой (document-storage + retrieval)**; не хватает **источника правды «кто какой tenant»** — то есть аутентификации и авторизации.

## 3. Вливание системы аутентификации в основную ветку

### 3.1 Зачем это нужно

- **Идентификация**: кто вызвал API (user/service).
- **Авторизация**: к каким tenant_id этот пользователь имеет доступ и с какими правами (только чтение, загрузка, удаление и т.д.).
- **Безопасность**: не доверять tenant_id из тела/формы, а выводить его из токена/сессии (или маппить user → допустимые tenant_id).

После вливания аутентификации:

- Gate (и при необходимости другие сервисы) проверяют токен/сессию;
- из токена извлекается **user_id** (и при необходимости **роль**);
- по user_id определяется список допустимых **tenant_id** (и опционально project_id);
- **tenant_id** в запросе либо подставляется из «профиля» пользователя, либо проверяется по списку разрешённых; подмена tenant_id клиентом блокируется.

Тогда мультитенантность становится не только «технической» (фильтр по полю), но и **безопасной** (доступ только к своим тенантам).

### 3.2 Что имеет смысл уточнить при консультации (в т.ч. с Мишей)

1. **Где живёт маппинг user → tenant(s)?**  
   Отдельный сервис/БД (Admin/Security в духе Ed-Fi ODS), существующий IAM, или таблицы в текущем Postgres (например, в document-storage или отдельная БД).

2. **Единая точка входа**: только Gate проверяет аутентификацию и подставляет/проверяет tenant_id, а document-storage и retrieval доверяют Gate (внутренняя сеть), или часть сервисов тоже должна понимать токены.

3. **Формат токена**: JWT (с tenant_id/roles в claims), сессия в Redis, или интеграция с внешним IdP (OAuth2/OpenID Connect). От этого зависит, как Gate извлекает tenant_id и права.

4. **Обратная совместимость**: период, когда tenant_id ещё передаётся явно для старых клиентов, и как постепенно перейти на «только из аутентификации».

5. **Роли и ACL**: текущее поле `acl` в метаданных — как оно будет стыковаться с ролями из auth (например, «только свои acl» или «роль администратора тенанта»).

## 4. Итоговая картинка: ODS → Multi-tenancy → Auth

```
[Клиент/UI]
     |
     | (будущее: токен + опционально явный tenant_id)
     v
[Gate]  ← здесь после вливания: проверка аутентификации, маппинг user → tenant_id
     |
     | tenant_id (из токена/сессии или проверенный)
     v
[document-storage]  ← ODS-слой: метаданные (Postgres) + файлы (S3), фильтр по tenant_id
     |
     | tenant_id в метаданных и запросах
     v
[doc-processor / retrieval]  ← индексация и поиск с tenant_id
```

- **ODS** — это наш операционный слой документов (document-storage + Postgres + потоки через Gate).
- **Multi-tenancy** уже есть на уровне данных и API; после вливания auth он станет контролируемым и безопасным.
- **Вливание системы аутентификации в основную ветку** закрывает вопрос «кто и к каким tenant_id имеет доступ» и позволяет не полагаться на переданный с клиента tenant_id без проверки.

Документ можно использовать как основу для обсуждения с Мишей и для планирования мержа ветки аутентификации в main.
